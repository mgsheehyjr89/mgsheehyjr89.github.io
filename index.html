Validation scope

Purpose And Evaluation Boundary

The objective is to determine whether compact GB10 clusters can support private, long-context coding and infrastructure agents with credible speed, memory headroom, and operational stability.

The current testbed consists of two privately funded ASUS GX10 / NVIDIA GB10 systems. The public evidence focuses on reproducible technical facts: node class, runtime configuration, model storage, benchmark data, kernel findings, known limitations, and the next validation gates.

The workload under evaluation is long-running agentic coding, not short-form chat. The target workload includes repository-scale context, multi-file edits, tool calls, test execution, error recovery, and parallel subagent activity. The relevant serving shape is therefore a single high-capability model with multiple slots and a large shared context pool.

Management and inference fabric remain separate. Administrative access, package work, sync, and recovery stay on the normal LAN. The high-speed link is treated as an isolated private fabric for model traffic and measurement.

Validation claims are separated by evidence level: measured single-node baselines, current two-node target configuration, and pending fabric/service results.

Limitations

Claims Not Yet Made

The following items remain open and must be completed before stronger two-node or cluster-level claims are published.

  1. 01
    Correct-speed QSFP fabric cable installed.

    The current temporary cable is sufficient for setup work but is not a basis for final GB10 fabric throughput claims.

  2. 02
    Forward and reverse fabric throughput recorded.

    Required evidence includes negotiated link speed, MTU, iperf3 both directions, and NIC error/drop/FEC counters.

  3. 03
    Long-context coding tasks run across model candidates.

    Qwen, MiniMax, and GLM candidates require the same repository-scale tasks, scoring rules, service geometry, and failure reporting.

  4. 04
    Repeated combined-service results captured.

    A validated two-node inference claim requires repeated service runs under one recorded launch configuration.

Current public status: two GB10 nodes are installed, model assets are staged, and single-node evidence is published. A final combined-inference-server claim is intentionally withheld until the fabric and service validation gates are complete.
