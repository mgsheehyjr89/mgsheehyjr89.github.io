Cable And Fabric Status

The current temporary cable is rated for 40 Gb/s. That is enough for setup work, interface configuration, and functional bring-up, but it is not enough to make a fair performance claim about the GB10 fabric.

The correct-speed cable is on the way. Until it is installed, I am not claiming final fabric bandwidth, final multi-node latency, or final model throughput.

  1. Confirm link negotiation without hard-coding the link to 40 Gb/s.
  2. Run forward and reverse iperf3 across the direct fabric.
  3. Run jumbo-frame checks if the negotiated path supports MTU 9000.
  4. Capture NIC error/drop/FEC counters before and after sustained load.
  5. Repeat model-serving benchmarks after fabric validation.

Every candidate will be tested on real coding tasks over long context. Short prompt demos are not enough.

Memory Budget

Each GB10 node has 128 GB of unified memory. I reserve 20 GB per node for the OS and system stability. That leaves roughly 208 to 216 GB combined for model weights, KV cache, compute scratch, graph buffers, runtime overhead, fragmentation, and safety margin.

The reserve is deliberately conservative. In internal testing, the system has tolerated lower free-memory levels, but the public benchmark should not depend on running the machines at the edge of OOM.

What A Third Node Unlocks

A third GB10 would move this from a two-node point-to-point test into a three-node fabric/ring validation. That matters because it creates a more realistic small-cluster topology and opens the door to model and context budgets that are not realistic on two nodes alone.

  • GLM-5.1 IQ3 class testing if the memory and runtime shape fit cleanly.
  • Qwen vs MiniMax vs GLM comparison under identical long-context coding tasks.
  • Ring/fabric behavior under sustained model traffic.
  • Failure-mode notes for agents that run for hours instead of minutes.
  • Power, thermal, memory, and stability reporting.
