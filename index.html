Build story

The Part That Matters

I am putting the narrative before the tables on purpose. The numbers matter, but the setup choices are what make the numbers useful to another builder, sponsor, or reviewer.

I started with two privately funded ASUS GX10 / NVIDIA GB10 systems because I wanted a straight answer to a practical question: can a small local cluster run coding agents with enough context to do real work, or does everything collapse into short prompts and half-finished edits?

The first rule was boring on purpose. I kept the management path separate from the high-speed fabric. SSH, package work, sync, and recovery stay on the normal LAN. The direct link is a private fabric for model traffic and measurements. If a fabric experiment goes sideways, basic administration should still be calm.

I am also not treating the temporary cable as a benchmark result. It is useful for bring-up. It is not the basis for a final cluster claim. The NICs should negotiate normally, and the correct-speed cable should be the point where forward and reverse fabric numbers become publishable.

The real test is not a one-prompt demo. It is a long-context coding workload with multiple agent slots, repo edits, tool use, test failures, queue behavior, and enough pressure to expose where the system actually spends time.

The model work is being framed the same way. Qwen3.5-397B IQ3 is the current two-node target because it fits the 8-slot, 4M-token working shape. MiniMax M2.7 IQ4 is staged because it may trade size for speed and coding quality. GLM remains the family I am watching for long-running agent behavior, but it does not get a winner label until it runs the same jobs under the same scoring rules.

Design choice

Management stays boring. Fabric work stays isolated. No default gateway moves to the experimental link.

Serving shape

Two nodes, one model, eight slots, and a four-million-token context pool for real agent/subagent work.

Public boundary

Private addresses, credentials, serials, receipts, raw shell history, and employer-private data stay out of the public pack.

Not claimed yet

The Next Validation Pass

The open work is concrete. It is not being blended into the verified numbers above.

  1. 01
    Install the correct-speed QSFP cable.

    Do not hard-code the link to the temporary 40G cable. Let the NICs negotiate, then record link speed, MTU, and counters.

  2. 02
    Run forward and reverse fabric tests.

    Capture iperf3 both directions, jumbo-frame checks when MTU 9000 is valid, and error/drop/FEC counters before and after sustained load.

  3. 03
    Run real coding tasks across the model candidates.

    Use long repositories, multi-file edits, tests, failures, and 8-slot subagent pressure. Score correctness, completion behavior, queue behavior, latency, decode speed, memory, and crashes.

  4. 04
    Publish only repeated combined-service results.

    A stronger two-node claim needs fabric numbers, long-context coding tasks, and repeated service runs under one recorded launch shape.

No public claim is being made yet that C-1 and C-2 are a fully validated combined inference server. The hardware is installed, the target is defined, and the missing tests are named.

A Third GB10 Changes The Experiment

With two nodes, the fabric test is point-to-point. With three, the topology becomes the small direct ring NVIDIA documents for Spark-class systems. That is a different validation problem, not just more memory.
