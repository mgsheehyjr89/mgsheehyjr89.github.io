Building a Two-GB10 Local AI Cluster

I bought the first two ASUS GX10 / NVIDIA GB10 systems myself because I wanted to stop guessing about local long-context coding agents. This is the setup work, the benchmark evidence, the mistakes, and the reason the next node matters.

Short version

I have two GB10-class systems online, large model assets staged, single-node benchmark data on disk, and a private fabric design ready for correct-speed validation. The next useful result is a repeated two-node service run under long-context coding pressure. After that, a third GB10 changes the topology from point-to-point into the small ring NVIDIA documents for Spark-class systems.

Why I Built It This Way

The workload I care about is not a chatbot answering one prompt. I want local agents that can stay inside a large repo, keep context, call tools, make edits, run tests, absorb failures, and finish without turning the hard parts into stubs.

That changes the whole shape of the system. The model has to be smart enough for coding, fast enough that agents do not stall, and large enough in context that several slots can work at once. I am targeting eight slots and a four-million-token context pool because the extra slots are for subagents, not for casual chat concurrency.

The main design rule is simple: never replace the boring management path with the experimental high-speed path. Add the fabric as a separate network and prove it before moving model traffic onto it.

The Network Shape

Both systems keep normal LAN management for SSH, updates, package sync, and recovery. The high-speed link is treated as a private fabric. It has no default gateway and no reason to carry ordinary management traffic.

I started with Ethernet mode because plain IP needs to be boring before RDMA, InfiniBand, or distributed runtime tuning enters the picture. The temporary cable is only useful for bring-up. It is not the basis for a final throughput claim, and I am not hard-coding the configuration to its temporary limit.

lspci | grep -Ei 'mellanox|nvidia|connectx|ethernet|infiniband'
ip -br link

sudo ethtool -i <fabric_iface>
sudo ethtool <fabric_iface>

# If Mellanox/NVIDIA tools are installed:
sudo mst status
sudo mlxconfig -d <device> q | grep -Ei 'LINK_TYPE|SRIOV|ROCE'

The Fabric Config

The direct fabric is a small private subnet with one address on each node and no gateway. The placeholders below are intentional. The useful public part is the shape, not private addressing.

# C-1
sudo nmcli con add type ethernet ifname <fabric_iface> \
  con-name gb10-fabric-c2 \
  ipv4.method manual \
  ipv4.addresses <c1_fabric_ip>/30 \
  ipv6.method disabled
sudo nmcli con mod gb10-fabric-c2 802-3-ethernet.mtu 9000
sudo nmcli con up gb10-fabric-c2

# C-2
sudo nmcli con add type ethernet ifname <fabric_iface> \
  con-name gb10-fabric-c1 \
  ipv4.method manual \
  ipv4.addresses <c2_fabric_ip>/30 \
  ipv6.method disabled
sudo nmcli con mod gb10-fabric-c1 802-3-ethernet.mtu 9000
sudo nmcli con up gb10-fabric-c1

The final fabric test is narrow: negotiated link speed, MTU, counters, forward iperf3, reverse iperf3, jumbo ping when MTU 9000 is valid, and error/drop/FEC counters before and after load.

iperf3 -s -B <c1_fabric_ip>
iperf3 -c <c1_fabric_ip> -P 8 -t 30
iperf3 -c <c1_fabric_ip> -P 8 -t 30 -R

ping -M do -s 8972 -c 3 <peer_fabric_ip>

sudo ethtool -S <fabric_iface> | grep -Ei 'err|drop|fec|crc|timeout|pause'

The Model-Serving Target

The question is not which file is biggest. The question is which single model gives the best coding intelligence at usable speed while leaving enough memory for KV cache, compute, and the operating systems.

Right now the working two-node target is Qwen3.5-397B IQ3 with eight slots sharing a four-million-token context pool. MiniMax M2.7 IQ4 is staged as a smaller challenger. GLM remains the family I am watching hardest for long-running agent behavior, but I am not calling a winner until the same coding tasks run across the candidates.

model: Qwen3.5-397B-A17B-UD-IQ3_S
slots: 8
context_pool: 4M tokens
kv_profile_observed: K=q8_0, V=turbo2
reserve: keep OS and service headroom on both nodes
test_load: real repos, multi-file edits, tests, failures recorded

Numbers I Can Publish Today

The strongest numbers today are still single-GB10 baselines and kernel/runtime findings. I am keeping two-node fabric claims separate until the correct cable and repeated service benchmarks are finished.

Measurement Recorded value Takeaway
GPT-OSS 120B tuned decode 63 tok/s batch-1 decode One GB10-class system can run a useful 120B lane.

What Broke Or Almost Broke

  • The cable mattered more than expected. A lower-rated cable is fine for setup, but it is not a final fabric benchmark.
  • The easy kernel toggles did not solve the bottleneck. MMQ-off stayed flat on decode and hurt prompt throughput; disabling stream-k globally was worse.
  • Benchmarks that look good in isolation are not enough. The real test is long-context coding with multiple agent slots under the same launch configuration.
  • Model quality claims are fragile unless Qwen, MiniMax, and GLM run the same tasks with the same scoring rules.

What I Would Repeat

  1. Name the nodes and document roles first.
  2. Keep LAN management separate from the high-speed fabric.
  3. Prove IP networking before RDMA or distributed runtime tuning.
  4. Record negative results because they stop the next person from wasting the same night.
  5. Publish benchmark data as CSV/JSON, not only screenshots.

Why A Third Node Changes The Work

With two nodes, the fabric test is point-to-point. With three, the topology becomes a small ring, which is the shape NVIDIA documents for three Spark-class systems using direct QSFP connections. That matters because the third node is not just more memory. It changes the validation: ring behavior, routing choices, service placement, failure behavior, and whether long-running agent traffic stays usable under load.

A fourth node would open Q3/Q4 tradeoff testing and redundancy experiments, but the first meaningful jump is two to three. That is why the sponsorship request starts with one permanent GB10-class node plus the right fabric cable.

The Next Concrete Test

  1. Install the correct-speed fabric cable.
  2. Record negotiated link speed, MTU, and counters.
  3. Run forward and reverse iperf3.
  4. Start the combined model service with the recorded launch shape.
  5. Run real long-context coding tasks across Qwen and MiniMax, then GLM if fit and runtime shape validate cleanly.
  6. Score correctness, completion behavior, queue behavior, time-to-first-token, decode throughput, memory pressure, and failures.
I am not trying to make a tiny cluster win a synthetic one-prompt chart. I am testing whether local GB10 systems can support the work commercial agents often fail at: staying on a task, using long context, running tools, editing real repositories, absorbing test failures, and finishing.
