Thera GB10 Validation Configuration Notes

This technical note records the current two-node GB10 validation shape: management isolation, direct-fabric configuration, benchmark evidence, runtime findings, model-serving target, and the gates required before stronger cluster claims are published.

Summary

The current testbed includes two ASUS GX10 / NVIDIA GB10-class systems, large model assets staged for side-by-side testing, and measured single-node inference data. The next publication-grade result is a repeated two-node service run after correct-speed fabric validation.

Validation Objective

The workload under evaluation is local long-context coding and infrastructure automation. The system must support repository-scale context, multiple agent slots, tool use, edits across files, test execution, failure recovery, and sustained operation without relying on hosted model infrastructure.

The two-node target is a single high-capability model served across both GB10 systems with eight slots and a four-million-token context pool. Additional slots are intended for subagent work, not general chat concurrency.

Design principle: keep routine management separate from the experimental high-speed fabric. Validate the fabric as an additional network, not as a replacement for recovery and administration.

Network Architecture

Both systems keep normal LAN management for SSH, updates, package sync, and recovery. The high-speed link is configured as an isolated private fabric with no default gateway. This keeps administrative recovery independent from fabric experiments.

The initial fabric mode is Ethernet/IP. RDMA and InfiniBand tuning are deferred until basic link negotiation, MTU behavior, counter health, and bidirectional throughput are demonstrated.

lspci | grep -Ei 'mellanox|nvidia|connectx|ethernet|infiniband'
ip -br link

sudo ethtool -i <fabric_iface>
sudo ethtool <fabric_iface>

# If Mellanox/NVIDIA tools are installed:
sudo mst status
sudo mlxconfig -d <device> q | grep -Ei 'LINK_TYPE|SRIOV|ROCE'

Fabric Configuration Shape

The direct fabric uses a small private point-to-point subnet with one address per node and no gateway. The placeholders below intentionally avoid publishing private addressing.

# C-1
sudo nmcli con add type ethernet ifname <fabric_iface> \
  con-name gb10-fabric-c2 \
  ipv4.method manual \
  ipv4.addresses <c1_fabric_ip>/30 \
  ipv6.method disabled
sudo nmcli con mod gb10-fabric-c2 802-3-ethernet.mtu 9000
sudo nmcli con up gb10-fabric-c2

# C-2
sudo nmcli con add type ethernet ifname <fabric_iface> \
  con-name gb10-fabric-c1 \
  ipv4.method manual \
  ipv4.addresses <c2_fabric_ip>/30 \
  ipv6.method disabled
sudo nmcli con mod gb10-fabric-c1 802-3-ethernet.mtu 9000
sudo nmcli con up gb10-fabric-c1

The temporary cable is acceptable for setup, but it is not a final performance basis. The validated run requires negotiated link speed, MTU, counter checks, forward iperf3, reverse iperf3, jumbo ping when applicable, and counter comparison before and after sustained load.

iperf3 -s -B <c1_fabric_ip>
iperf3 -c <c1_fabric_ip> -P 8 -t 30
iperf3 -c <c1_fabric_ip> -P 8 -t 30 -R

ping -M do -s 8972 -c 3 <peer_fabric_ip>

sudo ethtool -S <fabric_iface> | grep -Ei 'err|drop|fec|crc|timeout|pause'

Model-Serving Target

The selection question is the best single model for the two-node GB10 memory envelope when the objective is coding quality, inference speed, long context, and stable system headroom. Raw model size alone is not the selection criterion.

Current target: Qwen3.5-397B IQ3 across the two GB10 systems with eight slots sharing a four-million-token context pool. MiniMax M2.7 IQ4 is staged as a smaller challenger. GLM remains a high-interest candidate for long-running agent behavior, pending fit and runtime validation.

model: Qwen3.5-397B-A17B-UD-IQ3_S
slots: 8
context_pool: 4M tokens
kv_profile_observed: K=q8_0, V=turbo2
reserve: OS and service headroom on both nodes
test_load: real repositories, multi-file edits, tests, failures recorded

Current Benchmark Evidence

The strongest publishable values today are single-GB10 baselines and kernel/runtime findings. Two-node fabric and service claims remain gated on correct-speed cable validation and repeated combined-service runs.

Measurement Recorded value Interpretation
GPT-OSS 120B tuned decode 63 tok/s batch-1 decode One GB10-class system can run a useful 120B local inference lane.
GPT-OSS AIME25 local eval 0.925 over 240 samples Functional evaluation data accompanies throughput data.
Qwen Coder raw sweep 141.46 tok/s aggregate at npl=8 The backend has headroom before service overhead.
MMQ-off experiment Prompt throughput regressed from 162.40 to 66.96 tok/s Global MMQ disable is not an acceptable tuning path.

Known Limitations

  • Final fabric throughput is not claimed until the correct-speed cable is installed and tested.
  • Current public Qwen Coder graphs are single-GB10 historical baselines, not final combined-node service results.
  • Model quality claims require the same long-context coding tasks across Qwen, MiniMax, and GLM candidates.
  • Kernel tuning findings currently rule out broad switches; they do not yet constitute a deployable CUDA speedup.

Repeatable Practices

  1. Name nodes and document roles before distributed service work.
  2. Keep LAN management separate from high-speed fabric experiments.
  3. Validate plain IP networking before RDMA or distributed runtime tuning.
  4. Retain negative benchmark results when they materially narrow the tuning path.
  5. Publish benchmark data as CSV/JSON in addition to screenshots.

Three-Node Expansion Rationale

Two nodes validate point-to-point fabric behavior. Three nodes introduce the small direct-ring topology documented for Spark-class systems and enable validation of routing behavior, service placement, failure handling, and sustained agent traffic under load.

A fourth node enables higher-precision quantization tradeoff testing and redundancy experiments. The first materially different topology, however, begins at three nodes.

Next Validation Gate

  1. Install the correct-speed fabric cable.
  2. Record negotiated link speed, MTU, and interface counters.
  3. Run forward and reverse iperf3.
  4. Start the combined model service with the recorded launch shape.
  5. Run real long-context coding tasks across Qwen and MiniMax, then GLM if fit and runtime shape validate cleanly.
  6. Score correctness, completion behavior, queue behavior, time-to-first-token, decode throughput, memory pressure, and failures.
Validation target: determine whether local GB10 systems can sustain long-context coding agents that use tools, edit real repositories, recover from test failures, and complete multi-step work without relying on hosted model infrastructure.
